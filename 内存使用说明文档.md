# STM32H743 飞控工程内存使用说明

## 1. 文档目的

这份文档用于说明当前工程的内存分区、各内存区域的推荐用途，以及开发时应如何放置代码和数据。

结论先说：

- 当前工程的内存分区方案本身是正常的。
- 工程可以正常启动、正常运行。
- 真正决定系统是否稳定可靠的关键，不是“有没有分区”，而是“具体变量、缓冲区、代码段实际放到了哪一块内存”。

## 2. 当前工程实际生效的内存布局

当前构建使用的链接脚本是：

- `STM32H743VITX_FLASH.ld`

当前主要内存分区如下：

| 区域 | 起始地址 | 大小 | 当前属性 | 推荐用途 |
| --- | --- | --- | --- | --- |
| FLASH | `0x08020000` | `1920KB` | 可执行/只读 | 主程序代码、只读常量 |
| ITCMRAM | `0x00000000` | `64KB` | 高速、CPU直连 | 高频执行代码 |
| DTCMRAM | `0x20000000` | `128KB` | 高速、CPU直连 | 高频数据、关键控制数据 |
| RAM_D1 | `0x24000000` | `512KB` | AXI SRAM，当前设为 cacheable | 普通全局变量、RTOS 对象、普通缓存 |
| RAM_D2 | `0x30000000` | `288KB` | 当前 MPU 设为非缓存 | DMA 缓冲区优先放这里 |
| RAM_D3 | `0x38000000` | `64KB` | 当前 MPU 设为非缓存 | 低速共享数据、备用 DMA 区 |

## 3. 当前工程的段分配

在 `STM32H743VITX_FLASH.ld` 中，当前工程的主要段分配是：

- `.text / .rodata / 中断向量表` 放在 `FLASH`
- `.data / .bss / heap / stack` 默认放在 `RAM_D1`
- `.itcm_text` 放在 `ITCMRAM`
- `.dtcm_data` 放在 `DTCMRAM`
- `.dma_buffer` 放在 `RAM_D2`
- `.fast_stack` 放在 `DTCMRAM`

也就是说，工程已经预留了：

- 适合高速执行的 `ITCM`
- 适合关键数据的 `DTCM`
- 适合 DMA 的 `D2`

但是这些区域是否真正发挥作用，取决于你有没有显式把对象放进去。

## 4. 当前提供的内存放置宏

工程里已经提供了统一宏，文件位置：

- `Bsp/memory_section.h`

可用宏如下：

```c
#define ITCM_CODE  __attribute__((section(".itcm")))
#define DTCM_DATA  __attribute__((section(".dtcm")))
#define DMA_BUFFER __attribute__((section(".dma_buffer"))) __attribute__((aligned(32)))
#define FAST_STACK __attribute__((section(".fast_stack")))
```

推荐使用方式：

```c
ITCM_CODE void AttitudeLoop_Run(void)
{
    // 高频控制代码
}

DTCM_DATA static float pid_state[16];

DMA_BUFFER static uint8_t uart_rx_buf[256];
DMA_BUFFER static uint8_t sd_dma_buf[512];
```

## 5. 各内存区域推荐用法

## 5.1 FLASH

适合放：

- 普通代码
- 常量表
- 字符串常量
- 初始化只读数据

不适合放：

- 需要频繁修改的数据

## 5.2 ITCM

适合放：

- 高频执行函数
- 对延迟敏感的控制循环
- 中断中最关键的短函数

典型场景：

- 姿态解算核心
- 电机控制核心
- 高频滤波核心

不建议把大量普通业务代码都放进 ITCM，因为容量只有 `64KB`。

## 5.3 DTCM

适合放：

- 高频访问的小块关键数据
- PID 状态量
- EKF 中小规模频繁访问状态
- 控制回路临时工作数组

优点：

- CPU 访问非常快
- 适合对实时性要求高的数据

注意：

- 不要默认把所有大数组都扔进去
- 优先放“小而关键”的数据

## 5.4 RAM_D1

这是当前工程默认数据区。

适合放：

- 普通全局变量
- FreeRTOS 对象
- 普通任务数据
- 常规软件缓冲区

当前特点：

- 已开启 D-Cache
- 当前 MPU 配置下它是 cacheable

这意味着：

- CPU 访问效率较高
- 但如果某块数据要被 DMA 直接访问，就必须额外考虑 Cache 一致性

## 5.5 RAM_D2

推荐作为 DMA 专用区优先使用。

适合放：

- UART DMA 收发缓冲
- SPI DMA 缓冲
- SDMMC DMA 缓冲
- 未来如果接 ETH/USB DMA，也优先考虑这里

当前工程已经为它单独预留了：

- `.dma_buffer`

而且 `DMA_BUFFER` 宏已经做了 `32` 字节对齐，适合 Cortex-M7 的 Cache 维护。

## 5.6 RAM_D3

适合放：

- 不追求极限速度的共享数据
- 小型非缓存缓冲区
- 备用 DMA 区

一般优先级低于 `RAM_D2`，除非 `D2` 空间不够，或者有明确的外设访问规划。

## 6. 开启 D-Cache 后最重要的原则

当前工程在 `main.c` 中启用了：

- `SCB_EnableICache()`
- `SCB_EnableDCache()`

因此必须记住下面这条：

> 只要某块内存会被 DMA 直接读写，就不能只看“代码能跑”，必须考虑 Cache 一致性。

### 6.1 最稳妥的做法

把 DMA buffer 直接放进 `RAM_D2` 的 `.dma_buffer` 段：

```c
DMA_BUFFER static uint8_t imu_dma_rx[64];
DMA_BUFFER static uint8_t gps_dma_rx[256];
DMA_BUFFER static uint8_t sd_block_buf[512];
```

这样最省心，通常不需要额外做 D-Cache 清理/失效处理。

### 6.2 如果 DMA buffer 仍放在 D1

那就必须在合适时机做：

- `SCB_CleanDCache_by_Addr()`
- `SCB_InvalidateDCache_by_Addr()`

常见规则：

- CPU 写内存，DMA 要读之前：`Clean`
- DMA 写内存，CPU 要读之前：`Invalidate`

否则可能出现：

- DMA 读到旧数据
- CPU 读到旧数据
- 数据看起来“偶发错乱”
- 文件系统、串口、传感器数据偶发异常

## 7. 当前工程的实际建议

基于当前工程状态，建议按下面规则执行。

### 7.1 普通模块

默认保持现状即可：

- 普通代码放 `FLASH`
- 普通全局变量放 `D1`

### 7.2 控制核心

逐步把真正高频、短小、热点函数放到 `ITCM`：

- 姿态环
- 角速度环
- 电机混控

把关键状态量放到 `DTCM`：

- PID 状态
- 高频滤波状态
- 小型临时矩阵

### 7.3 DMA 相关模块

凡是会真正调用 DMA 收发的缓冲区，统一优先放 `DMA_BUFFER`：

- UART 接收环形缓冲
- SPI 传感器收发缓冲
- SDMMC 数据块缓冲

### 7.4 FatFs / SDMMC

这一项要特别注意。

当前工程的 `sd_diskio.c` 里虽然已经有 Cache Maintenance 的模板代码，但默认宏没有打开。如果后续实际使用 SD 卡 DMA，有两种方案：

方案 A：

- 将 SD 相关 DMA 缓冲固定放入 `RAM_D2`

方案 B：

- 继续使用 D1 buffer
- 但严格启用并维护 `Clean/Invalidate DCache`

优先建议方案 A，因为更直接、问题更少。

## 8. 什么时候必须显式指定内存位置

以下对象建议不要“随编译器默认放置”：

- DMA 缓冲区
- 高频控制核心函数
- 高频访问关键状态量
- 大型临时数组
- RTOS 中对实时性特别敏感的任务专用数据

一句话判断：

- 和实时性强相关：优先考虑 `ITCM/DTCM`
- 和 DMA 强相关：优先考虑 `D2/D3`
- 普通软件数据：默认放 `D1`

## 9. 推荐的使用规范

建议在团队内固定以下规范：

1. 所有 DMA buffer 必须显式使用 `DMA_BUFFER`
2. 高频控制代码如需优化，统一使用 `ITCM_CODE`
3. 高频关键数据统一使用 `DTCM_DATA`
4. 新增大数组前，先确认是否真的应该放在 `D1`
5. 只要开启 D-Cache，就不要把 DMA 数据当作普通变量处理

## 10. 当前方案的简化结论

当前工程的内存分区结论可以概括为：

- 分区方案本身正常
- 当前工程可以正常使用
- 但“分区存在”不等于“已经正确使用”
- 后续开发时，必须根据对象用途决定它放在 `ITCM / DTCM / D1 / D2 / D3` 中哪一块

最重要的一句话：

> 这个工程的问题不在“有没有分区”，而在“具体代码和数据有没有放到合适的分区里”。
